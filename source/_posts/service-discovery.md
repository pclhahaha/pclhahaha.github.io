### 1.1 注册中心核心能力

注册中心是微服务架构的通信中枢，所有服务间的调用最终都依赖它提供目标地址。其核心能力包括：

| 能力     | 说明                                                     |
| -------- | -------------------------------------------------------- |
| 注册     | 服务启动时向注册中心注册自身信息（IP/端口/元数据）       |
| 心跳     | 定期发送心跳维持租约，注册中心据此判断服务存活           |
| 注销     | 服务正常下线时主动注销；异常下线时心跳超时被动剔除       |
| 订阅     | 消费者订阅服务列表，注册中心推送变更通知                 |
| 通知     | 服务列表变更时通知所有订阅者（增量或全量）               |

**CAP 视角下的注册中心**：注册中心本质上是一个分布式协调系统。CAP 三者只能取其二——你在设计时必须做个选择：
- 要**一致性（CP）**：宁可短暂不可用，也不返回过期数据（Zookeeper）
- 要**可用性（AP）**：宁可返回过期数据，也保证服务正常响应（Eureka）
- 要**半 CP 半 AP**：既想保持一定的数据一致性，又要在分区时保证基本可用（Nacos）

### 1.2 Zookeeper

Zookeeper 使用 ZAB（ZooKeeper Atomic Broadcast）协议保证 CP——网络分区时牺牲可用性。当集群中 Leader 宕机，剩余节点需要重新选举（通常 30~120s），选举期间整个集群不可用。

**为什么 ZK 不适合大规模微服务注册？**
1. **写压力大**：每个服务实例定期心跳是一次写操作（创建临时节点），所有写都要通过 Leader 进行
2. **选主期间不可用**：服务无法注册/发现，新节点加入失败，老节点下线无法感知
3. **没有服务健康检查**：ZK 只关心连接是否存活，不关心实例健康状态。端口还活着但业务线程全阻塞，ZK 不会剔除。
4. **存储膨胀**：大量临时节点和 watcher 导致性能下降

**结论**：ZK 适合做分布式锁、配置管理、选主等**低频 + 强一致**场景，不适合做**高频 + 大量临时节点**的注册中心。

### 1.3 Nacos

Nacos 是阿里开源的"动态服务发现、配置和服务管理平台"，目标是同时支持服务发现（**CP+AP 混合**）和配置管理（**CP**）。

**架构层次**：

```
┌────────────────────────────────────────────────────────┐
│                       Nacos Server                      │
│  ┌─────────────────────┐  ┌──────────────────────────┐ │
│  │   服务发现模块 (AP)  │  │   配置管理模块 (CP)      │ │
│  │  - Distro 协议       │  │  - 简化 Raft 协议        │ │
│  │  - 最终一致性        │  │  - 强一致性              │ │
│  └─────────────────────┘  └──────────────────────────┘ │
└────────────────────────────────────────────────────────┘
         ▲     ▲                      ▲     ▲
         │     │                      │     │
    ┌────┴─┐ ┌─┴────┐           ┌────┴─┐ ┌─┴────┐
    │Svc A │ │Svc B │           │Svc A │ │Svc B │
    │注册   │ │订阅   │           │拉配置 │ │拉配置 │
    └──────┘ └──────┘           └──────┘ └──────┘
```

**Nacos AP 模式（Distro 协议）**：
- 临时实例（`ephemeral: true`）：基于 Distro 协议，节点间异步复制，分区时各节点独立提供服务，保证了 AP
- 心跳检测：客户端 5s 上报一次，15s 无心跳标记不健康，30s 剔除

**Nacos CP 模式（Raft 简化版）**：
- 持久实例（`ephemeral: false`）：基于 Raft 协议，选举期间不可写但可读，保证了 CP
- 适用于期望强一致的服务发现场景
