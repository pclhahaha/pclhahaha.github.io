### 1.1 业务特点与挑战

秒杀是电商系统中最极端的场景。痛点三角：

```
            瞬时极高并发
               /  \
              /    \
             /      \
    库存不超卖 ─────── 用户体验好（不卡、不白屏、不重复提交）
```

**核心挑战**：
- 瞬时流量可能是平时的 100~1000 倍
- 库存扣减必须在并发下绝对正确（少卖是损失，超卖是资损事故）
- 用户体感要流畅（页面快速响应，不能一直转圈）

### 1.2 容量估算

假设：10 万库存的 iPhone，1 秒内 100 万人同时抢购

```
峰值 QPS ≈ 1,000,000（实际集中在前 0.5 秒）

数据库 TPS：单机 MySQL 约 3000~5000 TPS
Redis 单机：10 万 QPS（读），8 万 QPS（写）

结论：流量不能直接打到数据库，必须在前面拦截 99% 以上的请求。
```

**分层拦截漏斗**：
```
100% 请求（100 万 QPS）
   │  前端：静态化 + 按钮置灰 + 防抖 ─→ 拦截 40%
   ↓
60 万 QPS
   │  CDN + 边缘限流 ─→ 拦截 30%
   ↓
30 万 QPS
   │  API 网关限流 + 用户维度排队 ─→ 拦截 80%
   ↓
6 万 QPS
   │  Redis 预减库存 ─→ 拦截 90%（库存只有 10 万）
   ↓
6,000 QPS
   │  MQ 异步排队写入 MySQL
   ↓
最终到达 DB：~3000 TPS（MySQL 可承受）
```

### 1.3 前端层优化

**静态化**
- 秒杀页面做成纯静态 HTML，提前推送到 CDN
- 唯一动态的是"库存数量"和"抢购按钮状态"，通过异步 JSONP/Ajax 获取
- 秒杀倒计时使用客户端时钟 + 服务端时间校准

**防刷与限流**
- 验证码滑块（人机识别），在抢购按钮点击时弹出
- 按钮防抖：点击后置灰 3 秒，禁止重复提交
- 前端随机延迟发送：在倒计时结束前 200ms~500ms 之间随机发送请求，削峰

**页面版本控制**
- 秒杀页面 URL 使用动态加签，每次活动生成不同的 URL
- 配合 CDN 的缓存策略，确保用户拿到最新页面
- 活动结束自动置灰页面按钮
